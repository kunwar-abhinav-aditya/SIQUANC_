To understand the QA process, consider the following example “What is the time zone of New York City?”.
An ideal QA system over DBpedia generates a formal representation of this question, that is a formal query (here expressed as SPARQL), which retrieves answers from the DBpedia endpoint. The query would look like: SELECT ?c dbr:New_York_City dbo:timeZone ?c. During this process, a QA system performs successive QA tasks.
In the first step (i.e. Named Entity Recognition (NER) and Named Entity Disambiguation(NED)), a QA system is expected to recognise and link the entities present in the question to its candidate(s) from DBpedia (i.e. mapping New York City to dbr:New_York_City). The next step is Relation Linking (RL), in which QA systems link the natural language predicate to the corresponding predicate in DBpedia (i.e. mapping time zone to dbo:timeZone). Finally, the Query Builder(QB) component obtains the linked resources and predicates from the previous steps to formulate a SPARQL query.
Sometimes it is also necessary to extract ontology classes in the question, so we have two components for the class linking task.
e.g. In the question “Which river flows through Seoul?”, river is a class which is expected to map to dbo:River.

The original idea of Frankenstein is described in the paper here. If you want to know more about Frankenstein as a resource platform, please see our another work here. Frankenstein integrates the components using the Qanary methodology.
In Frankenstein, you can run a complete QA pipeline (input the question, and getting the final answer). For this, you need at least a NED, a RL and a QB component. With the AGDISTIS NED component, you can also select any of the 11 NER components (for AGDISTIS, one of the inputs is recognised entities), and keep the same RL, and QB components.
You may have a requirement to use independent NER, NED, RL, or a CL component as well. In this case, just select a single component from any of the tasks, input your question or sentence and get the final output of the component. Frankenstein’s seamless UI will allow you to get your text annotated with minimal effort.
For example, if you got a list of 1000 tweets, in which you want to find the named entities and their respective DBpedia URLs, choose the component, upload the file to Frankenstein, and download the annotated results in .ttl format annotated in a formal ontology. Similarly you can use the platform for other tasks without any hassle.
